# 인터넷공룡 프로젝트 작업 기준 메모

## 0. 문서 목적

| 항목 | 내용 |
| --- | --- |
| 목적 | 인터넷공룡 홈페이지 작업 시 항상 참고해야 할 클라이언트 요구사항과 외주 프로젝트 운영 기준을 상기시키기 위한 문서 |
| 사용 대상 | 기획, 디자인, 개발, QA, 인수인계 작업 전반 |
| 사용 시점 | 새 작업 착수 전, 범위 변경 검토 시, 결과물 제출 전 |
| 프로젝트 성격 | 외주 제작물, 선수금 수령용 1차 납품 범위 포함 |

## 1. 이 프로젝트의 전제

| 항목 | 기준 |
| --- | --- |
| 프로젝트 성격 | 실제 서비스 런칭 완성본이 아니라 `선수금 수령용 1차 제작물`이다. |
| 우선순위 | 과한 고도화보다 `구조가 명확하고 납품 가능한 수준`이 더 중요하다. |
| 데이터 기준 | 1차는 `가상 데이터 기반`으로 구축하되, 실제 운영 전 `실제 데이터로 쉽게 교체 가능`해야 한다. |
| 운영 기준 | 실제 상담/가입 처리의 상당 부분은 오프라인 또는 전화 후속 프로세스를 전제한다. |
| 범위 통제 | 본인인증, 복잡한 정산, CRM 자동화, 실시간 요금 연동 등은 1차 범위가 아니다. |
| 문서 기준 | 모든 작업은 기획안, 디자인 방향서, 개발 명세서를 기준으로 한다. |

## 2. 클라이언트 요구사항 핵심 요약

| 구분 | 반드시 유지할 기준 |
| --- | --- |
| 서비스 목적 | 인터넷/TV/통신 가입 상담 및 상품 비교/신청 유도형 홈페이지 |
| 전환 구조 | 메인에서는 이름+연락처 중심의 간편 문의, 상품 페이지에서는 더 구체적인 문의 폼 |
| 디자인 톤 | 친근하지만 가볍지 않게, 캐릭터 느낌은 과하지 않게, 유치하지 않게 |
| UX 방향 | 비교가 쉽고, 문의로 자연스럽게 이어지는 구조 |
| 운영 기준 | 관리자 기능은 MVP만, 정책/법무/추적은 뼈대만 |
| 모바일 | 필수 대응 |
| 레퍼런스 활용 | 레이아웃과 흐름은 참고하되 그대로 복제하지 않는다 |
| 개발 원칙 | 실제 데이터 교체가 쉬운 구조, 유지보수 가능한 구조 |

## 3. 외주 작업 관점에서 반드시 지킬 기준

| 항목 | 기준 | 이유 |
| --- | --- | --- |
| 과설계 금지 | 1차 범위를 넘어서는 구조를 기본안에 넣지 않는다 | 일정/비용/합의 리스크 방지 |
| 설명 가능성 | 결과물은 클라이언트에게 쉽게 설명 가능해야 한다 | 외주는 이해 가능한 산출물이 중요 |
| 수정 용이성 | 카피, 상품, 후기, 혜택, 사업자 정보를 나중에 쉽게 바꿀 수 있어야 한다 | 실제 운영 전 변경 가능성이 높음 |
| 인수인계 가능성 | 다른 작업자가 이어받아도 구조를 이해할 수 있게 문서화한다 | 외주 특성상 담당자 변경 가능성 존재 |
| 숨은 운영비 최소화 | 1차부터 고정비/운영 복잡도가 높은 구조를 피한다 | 유지비 부담 최소화 |
| 범위 분리 명확화 | `1차 포함`과 `후속 고도화`를 항상 나눠서 설명한다 | 추가 요구 대응 기준 확보 |
| 리스크 선제 공지 | 막판 변경 위험이 있는 항목은 사전에 문서에 적는다 | 커뮤니케이션 분쟁 방지 |

## 4. 작업 시 기본 판단 기준

| 상황 | 판단 기준 |
| --- | --- |
| 기능을 추가하고 싶을 때 | “선수금 수령용 1차 납품에 꼭 필요한가?”를 먼저 본다 |
| UI를 복잡하게 만들고 싶을 때 | “모바일에서 바로 문의할 수 있는가?”를 먼저 본다 |
| 데이터 모델을 설계할 때 | “가상 데이터에서 실제 데이터로 쉽게 바꿀 수 있는가?”를 먼저 본다 |
| 관리자 기능을 늘리고 싶을 때 | “운영 핵심인가, 아니면 후속 편의 기능인가?”로 구분한다 |
| 기술 스택을 고를 때 | “빠르게 안정적으로 납품 가능하고, 인수인계가 쉬운가?”를 먼저 본다 |
| 문구/디자인을 정할 때 | “친근하지만 가볍지 않고, 신뢰형 상담 서비스처럼 보이는가?”를 먼저 본다 |

## 5. 범위 통제 기준

| 1차에서 반드시 포함 | 1차에서 제외 또는 후속 검토 |
| --- | --- |
| 홈, 상품 비교, 통신사 상세, 상품 상세, 후기, 게시판, 신청 페이지 | 본인인증 |
| 메인 문의 폼, 상품 상세 문의 폼, 신청 폼 | 실시간 설치 가능 조회 |
| 문의 저장, 관리자 로그인, 문의 관리 | CRM 자동 연동 |
| 상품/게시물 관리 MVP | 고급 권한 체계 |
| 정책 뼈대, 사업자 정보, 동의 기록 | 광고 추적 고도화 |
| 모바일 대응, 반복 CTA 구조 | 정산/결제/개통 처리 자동화 |

## 6. 산출물 품질 기준

| 영역 | 기준 |
| --- | --- |
| 기획 | 페이지 목적, CTA, 흐름, 운영 방식이 설명 가능해야 함 |
| 디자인 | 컬러/타이포/UI 시스템/모바일 구조가 시안 전 단계에서 합의 가능해야 함 |
| 개발 | 개발자가 바로 착수할 수 있는 라우트, 데이터 구조, 구현 우선순위가 있어야 함 |
| 관리자 | 최소한 문의 확인, 상태 변경, 메모, 상품/게시물 수정이 가능해야 함 |
| 문서 | 실제 클라이언트 제출용으로 읽어도 어색하지 않아야 함 |

## 7. 작업 전 체크

| 체크 항목 | 확인 |
| --- | --- |
| 현재 작업이 1차 범위 안에 있는가 | 필수 |
| 가상 데이터 기반이지만 실제 데이터 교체가 쉬운가 | 필수 |
| 모바일 전환 구조가 훼손되지 않는가 | 필수 |
| 클라이언트가 이해하기 쉬운 산출물 형태인가 | 필수 |
| 외주 특성상 인수인계/수정 용이성이 확보되는가 | 필수 |

## 8. 작업 중 금지 사항

| 금지 항목 | 이유 |
| --- | --- |
| 레퍼런스 화면을 그대로 복제 | 차별성 부족 및 외주 리스크 |
| 1차 범위를 넘는 기능을 기본 필수처럼 설계 | 일정/비용 증가 |
| 실제 운영 데이터가 없는데 실제 운영 전제 UI를 과하게 구성 | 납품 후 괴리 발생 |
| 관리자 기능을 CMS 수준으로 과도하게 확대 | MVP 원칙 위반 |
| 문서 없이 구현만 진행 | 외주 프로젝트 인수인계/수정 대응 불가 |
