## 0. 문서 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트명 | 인터넷공룡 홈페이지 1차 구축 |
| 브랜드명 | 인터넷공룡 |
| 서비스 성격 | 인터넷/TV/통신 가입 상담 및 상품 비교/신청 유도형 홈페이지 |
| 기획 기준 | 선수금 수령용 1차 제안서, MVP 중심 |

## 3. 벤치마크 해석

### 3-1. 레퍼런스 핵심 구조 해석

| 관찰 항목 | 레퍼런스 특징 | 해석 | 인터넷공룡 반영 방향 |
| --- | --- | --- | --- |
| 메인 진입부 | 상단에서 바로 이름+연락처 폼 제시 | 랜딩 직후 이탈 전에 리드 수집 | 메인 히어로에 즉시 문의 폼 고정 |
| 메뉴 구조 | 통신사별 상세, 신청서, 이벤트, 꿀팁, 후기 분리 | 상품 탐색과 신뢰 콘텐츠를 병행 운영 | 홈/상품비교/상품상세/혜택/가이드/후기 구조로 단순화 |
| 상품 제시 방식 | 통신사 대표 상품 카드 노출 후 상세 보기 유도 | 사용자가 상품 전체를 모르더라도 대표안으로 진입 가능 | 통신사별 “대표 추천안 1~2개” 우선 노출 |
| CTA 구조 | 페이지 곳곳에 `빠른견적`, `신청서 작성`, 전화상담 반복 | 어느 위치에서도 상담으로 회수 | 모든 주요 섹션 끝에 CTA 반복 배치 |
| 결합 추천 섹션 | 인터넷 단독/인터넷+TV/휴대폰 결합 등 시나리오 제안 | 사용자는 상품명보다 생활 시나리오로 이해 | 인터넷공룡도 “사용자 상황별 추천” 섹션 운영 |
| 사회적 증거 | 실시간 신청현황, 후기, 이벤트, 팁 게시판 | 신뢰와 활기 연출 목적 | MVP에서는 후기/가이드/혜택 게시판만 운영, 실시간 현황은 가상 데이터로 제한적 적용 |
| 상세 신청서 | 항목이 많아 실제 접수용으로 사용 가능 | 전환 전 단계에서는 부담, 후단계에서는 유용 | 메인 폼은 최소화, 상품페이지 폼만 상담에 필요한 추가 항목 수집 |
| 법무/정책 | 약관, 개인정보처리방침, 사업자 정보 노출 | 통신 가입상담 업종 특성상 신뢰 확보 필수 | 정책/사업자/동의문구는 MVP에서도 필수 포함 |

### 3-2. 벤치마크의 장단점 정리

| 구분 | 내용 | 인터넷공룡 기획 시 판단 |
| --- | --- | --- |
| 장점 | CTA가 매우 강하고 반복적이라 문의 유도가 명확함 | 유지 |
| 장점 | 통신사/결합유형/게시판이 분리돼 탐색 경로가 많음 | 유지하되 페이지 수는 축소 |
| 장점 | 후기/공지/팁이 신뢰 장치로 작동함 | 가상 콘텐츠 기반으로 1차 구축 포함 |
| 단점 | 정보량이 많아 첫 방문자에게 피로도가 큼 | 콘텐츠 밀도 축소 |
| 단점 | 신청서가 길어 초기 전환에 부담이 될 수 있음 | 메인 폼과 상품 상세 폼을 이원화 |
| 단점 | 실제 운영 기능이 많아 보이지만 운영 난이도가 높음 | 관리자 기능은 최소 운영 기준으로 제한 |

| 혜택/이벤트 상세 | 개별 혜택 설명 | 내용 본문, 적용 대상, 유의사항, 하단 문의폼 | `상담 신청` | 운영자가 수정 가능 |
| 가입 가이드 목록 | 정보성 콘텐츠 탐색 | 카드 목록, 인기 글, FAQ | `상품 비교하기`, `문의하기` | SEO/신뢰 목적 |
| 가입 가이드 상세 | 가입 이해도 향상 | 본문, 관련 상품 연결, 하단 문의폼 | `빠른 문의` | 가상 콘텐츠 |
| 후기 목록 | 신뢰 강화 | 후기 리스트, 베스트 후기, 하단 폼 | `상담 문의` | 작성 기능은 관리자 등록형으로 제한 가능 |

| 관리자 문의 관리 | 리드 처리 | 목록, 검색/필터, 상세, 메모, 상태 변경 | `저장`, `상태 변경` | 핵심 운영 페이지 |

| 관리자 게시판 관리 | 콘텐츠 운영 | 혜택/가이드/후기 목록, 등록/수정 | `등록`, `수정` | 에디터는 단순 텍스트/이미지 수준 |

## 6. 사용자 흐름

### 6-1. 핵심 전환 흐름

| 흐름명 | 사용자 행동 | 시스템/페이지 반응 | 전환 포인트 |
| --- | --- | --- | --- |
| 빠른 문의 흐름 | 홈 진입 → 이름/연락처 입력 → 동의 체크 → 제출 | 완료 페이지 이동, 관리자 문의 저장, 상담 대기 상태 생성 | 메인 문의 접수 |
| 상품 탐색 흐름 | 홈 또는 상품비교에서 대표 상품 클릭 → 상품 상세 확인 → 상세 문의 폼 작성 | 상품 식별 정보와 함께 문의 저장 | 상품 맥락이 있는 리드 확보 |
| 신뢰 콘텐츠 흐름 | 후기/가이드/이벤트 읽기 → 하단 CTA 클릭 → 문의 폼 제출 | 읽은 콘텐츠 소스값 함께 저장 | 정보 탐색형 방문자 회수 |
| 전화 직행 흐름 | 어느 페이지에서든 전화 CTA 클릭 | 전화 연결 또는 번호 복사 | 즉시 상담 유도 |

### 6-2. 운영 흐름

| 단계 | 관리자 행동 | 처리 내용 |
| --- | --- | --- |
| 1 | 문의 접수 확인 | 신규 문의 상태 `신규`로 저장 |
| 2 | 1차 연락 | 유선 상담 후 상태를 `연락완료` 또는 `부재`로 변경 |
| 3 | 상담 분류 | 희망 통신사/상품/지역/설치예정 여부 메모 기록 |
| 4 | 후속 진행 | 실제 신청은 전화/메신저/오프라인 프로세스로 별도 처리 |
| 5 | 종료 | `상담완료`, `신청완료`, `보류` 중 하나로 종료 |

## 7. 공통 문의 폼 + 상품 상세 문의 폼 설계

### 7-1. 공통 문의 폼

| 항목명 | 타입 | 필수 | 설명 | 검증/처리 |
| --- | --- | --- | --- | --- |
| 이름 | 텍스트 | 필수 | 상담 호칭 및 문의 식별용 | 2자 이상 |
| 연락처 | 휴대폰 | 필수 | 상담 회신용 | 숫자/하이픈 정규화 |
| 개인정보 수집 동의 | 체크박스 | 필수 | 상담 접수 법적 동의 | 미체크 시 제출 불가 |
| 유입 페이지 | 숨김값 | 자동 | 어느 페이지에서 접수됐는지 기록 | URL/페이지명 저장 |
| 상품 식별값 | 숨김값 | 자동/선택 | 상품 상세에서 제출 시 상품명/상품코드 저장 | 관리자 조회용 |
| UTM/리퍼러 | 숨김값 | 자동 | 마케팅 추적 뼈대 | 값이 없으면 null 저장 |
